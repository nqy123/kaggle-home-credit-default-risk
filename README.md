# Kaggle Home Credit 违约风险预测：多表关系特征工程框架

![Result Card](assets/result_card.png)

## 这个项目在简历里的定位

这是一个典型金融风控项目，亮点是多表关系建模。当前仓库已经整理好训练脚本和特征框架，适合作为后续继续冲分的风控建模底座。

## 业务问题与评价指标

利用申请主表、征信历史、历史申请、POS、分期还款和信用卡行为预测贷款违约概率，指标为 ROC-AUC。业务上对应贷前风控和信用评分。

当前项目已发布代码框架，尚未记录正式线上提交分数。预期输出为 `outputs/submission_lgbm_best.csv`。

## 我的整体解决思路

我把这个项目拆成五步：

1. **先理解数据**：看目标分布、字段类型、缺失模式、train/test 差异和指标特点。
2. **再做清洗**：把明显的脏值、哨兵值、语义缺失和泄露风险处理干净。
3. **重点做特征工程**：不是机械造列，而是把业务直觉、数据结构和模型能力连接起来。
4. **用 OOF 做验证**：每个模型都尽量通过交叉验证产生 OOF，避免只看一次线上提交。
5. **最后复盘取舍**：线上不涨的复杂方案不保留，最终只保留稳定、能解释、能复现的版本。

## EDA 与数据理解

- Home Credit 的难点不是单表字段多，而是客户历史行为分散在多张关系表中。
- 违约风险与偿债能力、历史逾期、申请被拒、还款延迟、信用卡使用率都有关系。
- 多表数据量大，内存优化和聚合粒度直接影响能否顺利训练。

## 数据清洗策略

- 将 `DAYS_EMPLOYED=365243` 这类哨兵值还原为缺失。
- 对大表做 reduce_mem，降低训练时内存压力。
- 清理列名特殊字符，避免 LightGBM 因列名报错。

## 特征工程：不是堆字段，而是写入业务逻辑

| 特征设计 | 为什么这样构造 | 预期收益 |
| --- | --- | --- |
| 信用/收入比例 | 贷款金额相对收入越高，偿债压力越大。 | 刻画负债压力。 |
| 年金/收入比例 | 月供或年金占收入比例是风控核心指标。 | 衡量还款能力。 |
| bureau 聚合 | 历史信贷账户能反映长期信用行为。 | 捕捉历史负债和逾期风险。 |
| previous approved/refused | 过去申请是否被拒能反映机构对客户的历史判断。 | 补充申请质量信号。 |
| installment late payment | 真实还款延迟比申请信息更接近违约行为。 | 建模行为风险。 |
| credit card utilization | 信用卡额度使用率高可能代表资金压力。 | 识别流动性风险。 |

这部分是整个项目最值得讲的地方。我的处理原则是：**先解释变量为什么可能影响目标，再决定把它做成比例、计数、交叉、分箱、rank、文本向量还是序列语义表示**。这样做出来的特征不是“玄学调参”，而是有业务含义、有验证闭环的建模资产。

## 模型选择：为什么用这些模型

- LightGBM 适合大规模稀疏聚合特征，是 Home Credit 经典强模型。
- 5-fold StratifiedKFold 保持违约比例稳定。
- 提供 full relational FE 和 GPU fast 两个入口，兼顾效果和运行成本。

我没有把模型当成黑箱堆叠，而是根据数据形态选择模型：结构化数据优先树模型，类别特征多时重视 CatBoost，高维匿名数值用 LightGBM，文本任务则先用 TF-IDF 建强 baseline，再用 Transformer 补语义理解。

## 验证与防止过拟合

- 使用 OOF 或交叉验证观察本地泛化表现。
- 区分 public/private 分数，避免过度追逐单次 public leaderboard。
- 严格排除 ID、target、后竞赛标签等泄露来源。
- 对融合方案做线上复盘：如果复杂方案不涨分，就回退到更稳定版本。

## 运行结果分析

- 这个项目的价值在于风控思路完整：申请信息只是表层，历史行为才是关键。
- 多表聚合把客户过去的信用轨迹压缩成模型可学习的风险画像。
- 下一步需要先跑出线上分数，再根据 feature importance 和残差继续迭代。

## 如何复现

安装依赖：

```bash
pip install -r requirements.txt
```

复现时先从 Kaggle 下载原始数据到项目约定的数据目录。部分仓库为了保持轻量，只保留最佳提交文件、实验日志和核心说明；如果仓库中存在 `src/`、`notebooks/` 或 `kaggle_kernel_*`，优先从这些入口运行训练。

常见入口示例：

```bash
python src/train_best.py
# 或在 Kaggle 上运行 kaggle_kernel_* 中的 GPU kernel
```

如果当前项目只保留了最佳产物，可直接查看 `outputs/` 中的 OOF、prediction、submission 和实验摘要文件。
